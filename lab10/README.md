4. **Реалізація стійкого зв'язку (LWT та QoS):**
   Для глибинного тестування мережевої поведінки протоколу в Node-RED було створено два окремі брокери з унікальними ідентифікаторами клієнтів (Client ID):
   * **Видавець (`client_v16_pub`):** Налаштовано повідомлення "Останньої волі" (Will Message): топік `artem/client1/status`, корисне навантаження `offline (break)`, QoS = 1, Retain = true. Повідомлення про появу в мережі (Birth) — `online`.
   * **Приймач (`client_v16_sub`):** Налаштовано на постійну сесію шляхом **вимкнення** прапорця **Clean Session**. Рівень QoS для топіка `artem/device1/random` виставлено на `1`.

5. **Тестування функції LWT:** При аварійному вимкненні Node-RED (імітація обриву зв'язку шляхом відключення мережі та зупинки процесу `node-red` в консолі за допомогою `Ctrl + C`), брокер Mosquitto через Keep-Alive інтервал автоматично розіслав іншим підписникам статус `offline (break)`, що зафіксовано в MQTT Explorer.

6. **Тестування черги повідомлень (QoS = 1):** Клієнт-приймач `client_v16_sub` був програмно переведений в офлайн (режим *Disabled*). Протягом 2 хвилин видавець продовжував надсилати генерацію чисел. Після активації приймача назад у мережу (режим *Enabled*) та виконання *Deploy*, брокер Mosquitto миттєво передав накопичений буфер даних лавиною в одну секунду, підтверджуючи збереження повідомлень з QoS = 1 для постійних сесій.

## Висновки
Під час виконання лабораторної роботи було детально вивчено технологію MQTT та принципи побудови IoT-систем на її основі. На практиці доведено ефективність використання механізму **LWT (Last Will)** для контролю працездатності віддалених датчиків та контролерів в системах промислової автоматизації. Також перевірено надійність збереження даних за допомогою комбінації **QoS = 1** та відключення **Clean Session**, що дозволяє IoT-архітектурі функціонувати без втрати важливих технологічних параметрів навіть при тривалих збоях у мережі зв'язку.
